Quickstart

Go from a fresh download to your first autonomous threat response in under 15 minutes. This guide assumes you've met the system requirements and installed Docker, CUDA drivers, and Wazuh.

i
AI SOC is fully air-gapped. Once the container images are present on the host, no step in this guide requires internet access.

1 · Bring up the stack

From the unpacked release directory, start the full Docker Compose stack. This launches Kafka, Redis, PostgreSQL, the vLLM inference server, and ChromaDB.

bash
docker compose up -d
# ✓ kafka      started
# ✓ redis      started
# ✓ postgres   started
# ✓ vllm       started  (loading Qwen2.5-72B-AWQ…)
# ✓ chromadb   started

Wait for the model to finish loading, first boot can take a few minutes while weights are mapped into GPU memory. Check readiness:

bash
docker compose logs -f vllm | grep ready
"vLLM API server ready on 0.0.0.0:8000"

2 · Configure the environment

Copy the example environment file and point the Wazuh bridge at your SIEM. All values stay local.

yaml - .env
WAZUH_API_URL=https://wazuh.internal:55000
WAZUH_API_USER=soc-bridge
KAFKA_BROKERS=localhost:9092
VLLM_ENDPOINT=http://localhost:8000/v1
CONFIDENCE_THRESHOLD=0.85
HITL_ENABLED=true
SOAR_WEBHOOK=http://shuffle.internal/api/v1/hooks/ingest
!
Keep CONFIDENCE_THRESHOLD at 0.85 or higher in production. Lowering it increases autonomous actions and reduces the share of alerts routed to human review.

3 · Watch the first alert

Start the triage loop and stream live decisions to your terminal. Open the Electron UI in parallel to see the HITL dashboard populate.

bash
ai-soc triage --watch
[ingest]   alert id=a91f host=host-04 rule="credential dumping"
[enrich]   mapped → MITRE T1003 · tactic=Credential Access
[decide]   confidence=0.91 → AUTO-RESPOND
[respond]  playbook #44 · isolate host-04, rotate creds
[done]     resolved in 2.3s

That's it, the pipeline ingested a Wazuh detection, enriched it with MITRE context, scored confidence above the threshold, and executed a SOAR playbook autonomously.

Pipeline overview

AI SOC is a streaming pipeline. Wazuh detections enter Kafka, eight LangGraph agents process them in stages, and approved decisions trigger Shuffle SOAR workflows. Redis holds working state; PostgreSQL stores the audit log.

text
Wazuh ──▶ Kafka ──▶ [ ingest → triage → enrich → decide → respond ] ──▶ Shuffle SOAR
                         │                          │
                      ChromaDB (RAG)            PostgreSQL (audit)

The 8 agents

Each agent is a node in the LangGraph state machine with a single responsibility:

  1. Ingest: normalizes Wazuh alerts into the pipeline schema.
  2. Dedupe: collapses duplicate and correlated detections.
  3. Triage: assigns initial severity and priority.
  4. Enrich: attaches MITRE ATT&CK techniques and adversary context.
  5. Retrieve: pulls relevant runbooks from ChromaDB (RAG).
  6. Decide: scores confidence and chooses auto-respond or HITL.
  7. Respond: executes the Shuffle SOAR playbook.
  8. Report: writes the audited outcome to PostgreSQL.

API Reference

The FastAPI backend exposes REST and WebSocket endpoints. Submit an alert for triage:

bash
curl -X POST http://localhost:8080/v1/alerts \
  -H "Content-Type: application/json" \
  -d '{"rule":"T1059","host":"host-02","severity":"med"}'

# -> { "id": "a92c", "decision": "HITL", "confidence": 0.62 }

Deployment

For multi-node deployments, run the inference tier on GPU hosts and the agent tier on CPU hosts, connected by a shared Kafka cluster. See the requirements page for sizing guidance.

Troubleshooting

vLLM fails to start

Confirm the NVIDIA container toolkit is installed and nvidia-smi reports your GPU inside a container. The 72B-AWQ model requires at least 48 GB of VRAM across visible devices.

No alerts appearing

Verify the Wazuh bridge can reach your SIEM and that WAZUH_API_USER has read access to the alerts index. Check docker compose logs wazuh-bridge.
